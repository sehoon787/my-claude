# Refactoring Prompt

다음 순서대로 이 프로젝트를 리팩토링해줘. 각 단계마다 해당 skill을 실행해.

## 1단계: 분석 및 계획

/plan
- 현재 코드베이스 구조를 파악하고 리팩토링 범위와 우선순위를 정리해줘.
- 네이밍 컨벤션(camelCase/PascalCase/snake_case 등)이 일관되지 않은 부분을 찾아줘.
- 재사용 가능하지만 중복된 코드, 불필요한 의존성, 죽은 코드를 식별해줘.
- 구현 전 반드시 내 승인을 받아.

## 2단계: 코딩 표준 적용

/coding-standards
- 업계 표준 네이밍 컨벤션으로 통일해줘.
- 파일/폴더 구조가 역할별로 명확히 분리되도록 모듈화해줘.
- 단일 책임 원칙(SRP)을 지키고, 적절한 디자인 패턴을 적용해줘.

## 3단계: 백엔드 리팩토링

/backend-patterns
- API 라우트, 서비스 레이어, 데이터 액세스 레이어를 명확히 분리해줘.
- 에러 핸들링, 입력 검증, 미들웨어 구조를 개선해줘.
- DB 쿼리 최적화와 N+1 문제 등 성능 이슈를 해결해줘.

## 4단계: 프론트엔드 리팩토링

/frontend-patterns
- 컴포넌트를 재사용 가능한 단위로 분리해줘.
- 상태 관리, 데이터 페칭, 렌더링 최적화를 개선해줘.
- UI/비즈니스 로직 분리를 확실히 해줘.

## 5단계: 죽은 코드 제거

/refactor-cleaner 에이전트를 사용해서 미사용 코드, 중복, 불필요한 의존성을 제거해줘.

## 6단계: 보안 점검

/security-review
- 사용자 입력, 인증, API 엔드포인트, 민감 데이터 처리에 취약점이 없는지 확인해줘.

## 7단계: TDD 기반 검증

/tdd-workflow
- 리팩토링된 모든 모듈에 대해 테스트를 먼저 작성하고 통과시켜줘.
- 단위/통합 테스트 커버리지 80% 이상을 목표로 해줘.
- 기존 동작이 깨지지 않았음을 검증해줘.

## 핵심 원칙

- 기존 동작을 절대 변경하지 말 것 (리팩토링만)
- 매 변경마다 테스트로 검증
- 과도한 추상화 금지, 현재 필요한 만큼만
- 커밋은 단계별로 나눠서 진행
